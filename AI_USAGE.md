## 2. AI 활용 전략

### 컨텍스트 파일 운용

- `PRD.md`: 스키마, confidence 룰, 파이프라인 명세 등 확정된 설계 전체를 담은 파일. Claude Code가 매 작업 시작 시 반드시 읽도록 지시함으로써 "설계 의도와 다른 코드 생성"을 방지
- `PRD_ver02.md`: 구현 과정에서 변경된 사항(LLM 모델, 파일명 컨벤션, 프롬프트 개선 등)을 누적 반영한 버전

### 프롬프트 원칙

- **PRD 준수 강제**: 모든 구현 지시에 "PRD에 없는 내용이 추가되면 무조건 먼저 확인을 받아야 해"를 명시적으로 포함. AI가 임의로 설계를 확장하는 것을 방지하고, 추가 여부를 내가 직접 판단
- **단계별 검증**: 파일 하나씩 구현 → 실행 → 결과 확인 → 다음 파일 진행. AI 결과물을 일괄 수용하지 않고 단계마다 검토
- **중단 조건 명시**: "에러 발생 시 무조건 중단하고 보고" 지시. AI가 오류를 자의적으로 우회하지 못하도록 제어

## 3. 내가 직접 판단하고 수정한 사례

### ① 파일명 컨벤션 설계 — CLI 인자 방식 전면 폐기

**AI가 제안한 것:** `python main.py --title "노바드림 회의" --advertiser "노바드림" --date "2026-06-01"` 형식의 CLI 인자 방식

**내가 판단한 것:** 100명 사원이 매 회의마다 CLI 인자를 직접 입력하는 것은 현실적이지 않다. 파일명 자체가 메타데이터 역할을 해야 운영 가능한 시스템이 된다.

**내가 결정한 방향:** `{advertiser}_{YYYYMMDD}_{title}.json` 파일명 컨벤션으로 전환. 파일명에서 광고주·날짜·제목을 자동 파싱하도록 설계 변경

**결과:** main.py에서 CLI 인자 3개 제거, `_parse_filename_metadata()` 자동 파싱 함수로 대체

### ② ABBREVIATION_DICT / VAGUE_PATTERNS 직접 확장

**AI가 제안한 것:** PRD 명세대로 ABBREVIATION_DICT 8개, VAGUE_PATTERNS 8개

**내가 판단한 것:** PRD 기본값은 일반적인 케이스만 커버한다. 실제 제공된 transcript를 직접 읽으며 등장하는 표현들을 확인했고, 실무에서 더 자주 쓰이는 광고 용어와 실제 발화 패턴이 누락되어 있음을 발견

**내가 결정한 방향:** transcript의 각 발화 id를 직접 분석하여 실제 등장 표현(id:19 "했었나", id:35 "맞죠?", id:12 "아닌가" 등)을 파악하고 확장 목록을 직접 작성해 교체 지시

**결과:** ABBREVIATION_DICT 8개 → 23개 (CPC, CPV, ROI, KPI 등 추가), VAGUE_PATTERNS 8개 → 21개

### ③ LLM 모델 선택 — 쿼터 현실에 맞는 모델 탐색

**AI가 제안한 것:** 처음에는 Claude API(Anthropic) 사용. 이후 Gemini 2.0 Flash로 변경 제안

**내가 판단한 것:** 무료 티어 환경에서 실제 API 호출이 동작하는 PoC를 만들어야 한다. 모델 선택은 기술적 성능뿐 아니라 쿼터 제약을 고려해야 한다.

**내가 결정한 방향:** API 호출 테스트를 통해 직접 사용 가능한 모델을 탐색. gemini-2.0-flash(limit:0) → gemini-flash-latest → gemini-2.5-flash 순으로 실제 동작 확인 후 결정. 쿼터 소진 시에도 파이프라인이 crash되지 않도록 예외 처리 보완 요청

### ④ 대시보드 전면 재구성 — "의사결정자 관점" 직접 설계

**AI가 제안한 것:** PRD 명세대로 차트 4개를 순서대로 나열한 구조

**내가 판단한 것:** DB에 적재된 실제 데이터를 직접 조회해보니, 현재 대시보드로는 "회의 직후 무엇을 해야 하는가"를 한눈에 파악할 수 없었다. 차트 나열은 분석가용이지, 의사결정자용이 아니다.

**내가 결정한 방향:** 구체적인 새 구조를 직접 설계해 지시
- KPI 카드 4개 → 현황 즉시 파악
- 전체 액션아이템 테이블 → 담당자 필터 + 중복 의심 자동 감지
- 상단 KPI → 전체 조감 → 세부 분석으로 흐름 구성

**결과:** dashboard/app.py 전면 재작성(v2). 위젯 4개 → 5개, 의사결정 흐름 구조로 재편

### ⑤ 시각화 반복 개선 — 직접 보고 판단

**AI가 제안한 것:**
- confidence 분포: 10구간 히스토그램 (0.0~0.1, 0.1~0.2...)
- 주차별 추이: 두 지표를 같은 Y축의 라인 차트
- 주차 레이블: "6/1" 형식

**내가 판단한 것:**
- 히스토그램 구간이 너무 세분화되어 의사결정자가 "몇 건을 검수해야 하는가"를 즉시 파악할 수 없음
- 회의 수(1)와 액션아이템 수(18)를 같은 축에 그리면 회의 수 선이 바닥에 붙어 보이지 않음
- "6/1"은 날짜인지 비율인지 직관적으로 알 수 없음

**내가 결정한 방향:**
- 신호등 카드 3개(저/중/고신뢰) + 3구간 바 차트로 변경 → "몇 건 검수 필요"가 즉시 보임
- 중복 의심 항목도 검수 드릴다운에 통합
- 독립 Y축을 가진 쌍막대 차트로 변경 → 두 지표 비교 명확
- "6월 1주차" 형식으로 변경 → 주차 의미 명확

### ⑥ deadline 날짜 추론 개선 — 데이터 직접 확인 후 문제 정의

**AI가 제안한 것:** deadline을 원문 그대로 저장 ("내일 오전", "수요일 오전까지")

**내가 판단한 것:** DB에 저장된 실제 데이터를 직접 조회했을 때, "수요일 오전까지"라는 표현이 실제로 언제인지 알 수 없었다. 파일명에 회의 날짜가 있으므로 이를 기준으로 LLM이 실제 날짜를 추론할 수 있다.

**내가 결정한 방향:** meeting_date를 LLM 프롬프트에 함께 전달, "내일 오전" → "2026-06-02 오전 (내일 오전)" 형식으로 추론하도록 프롬프트 개선 지시

### ⑧ 화자 분리 미적용 결정

**AI가 제안한 것:** pyannote.audio 라이브러리로 화자 분리 구현 가능 (가산점 선택 사항)

**내가 판단한 것:** PoC 단계에서 화자 분리는 HuggingFace 토큰 설정, 모델 다운로드 등 진입 장벽이 높고, 제공된 transcript JSON에는 이미 화자 정보가 포함되어 있다. 복잡도 대비 효용이 낮음.

**내가 결정한 방향:** STT 모드에서는 SPEAKER_00 고정으로 단순 처리. transcript JSON 모드(주 사용 경로)는 이미 화자 분리된 데이터를 활용

### ⑨ LLM 추출 품질 개선 — 검증 후 직접 판단한 프롬프트 3가지 수정

**내가 판단한 것:** DB에 적재된 실제 추출 결과(24건)와 transcript 37개 발화를 직접 대조하여 세 가지 구조적 문제를 발견했다.
1. 회의 아젠다 발화(id:2 "봐야 할 것 같아요")가 액션아이템으로 오추출 — 4건
2. 같은 작업이 청크 경계에서 중복 추출 — 3쌍(6건)
3. 수행 중인 행동의 발화자가 담당자로 매핑 안 됨 — 1건

**내가 결정한 방향:** 세 가지를 직접 지시
- 추출 기준 명시: 확정 약속("~할게요")만 추출, 아젠다·예측 표현은 제외
- 부정 few-shot 추가: 수아의 아젠다 발화 → `[]` 반환 예시
- 담당자 추론 규칙 추가: "요청했는데" = 발화자가 담당자
- 청크 간 컨텍스트 전달: 이전 청크 추출 결과를 다음 청크 프롬프트에 포함해 중복 방지

**결과:** 24건 → 11건. 아젠다 오추출 제거, 중복 제거, 담당자 정확도 향상

---

### ⑩ 재확인 확정 표현 추출 기준 추가 — 맥락 기반 판단

**AI가 제안한 것:** 추출 기준 개선 후 재검증하여 "제안서 목요일 마감" 1건이 누락됨을 확인. "~거니까" 표현 추가를 제안.

**내가 판단한 것:** "~거니까"를 추가하면 범위가 너무 넓어진다. 핵심은 단어 형태가 아니라 **"그대로"라는 맥락** — 이미 합의된 사항을 재확인하는 발화가 액션아이템이 되어야 한다. "목요일 오후 그대로 가는 거니까"에서 중요한 것은 "그대로"이다.

**내가 결정한 방향:**
- 추출 기준에 `재확인 확정` 카테고리 신설: "~그대로", "~그대로 가는 거", "~변동 없음"
- 단, "그대로"가 있어도 단순 상황 설명이면 제외한다는 단서 추가
- few-shot 예시 6 직접 작성: `"제안서는 어쨌든 목요일 오후 그대로 가는 거니까"` → 제안서 제출 액션

---

### ⑪ 모델 종류에 따른 추출 품질 편차 — 모델 일관성의 중요성 확인

**내가 판단한 것:** Gemini API 무료 티어 쿼터(일 20회)가 모델마다 별도로 적용되어, 하나의 모델이 소진되면 다른 모델로 전환해야 했다. 이 과정에서 **같은 프롬프트를 사용해도 모델마다 결과가 다르다**는 사실을 실험적으로 확인했다.

- gemini-3.1-flash-lite: 컨텍스트 전달을 잘 반영해 중복 0쌍, 11건
- gemini-3.1-flash-lite-preview: 컨텍스트를 덜 엄격하게 반영해 중복 2쌍 재발, 14건

**결론:** 실제 운영 시 모델을 고정하지 않으면 추출 결과의 일관성을 보장할 수 없다. 유료 플랜으로 단일 모델을 고정하는 것이 필수다.

## 5. 실험 결과와 한계점

### 5.1 AI 활용 방식

Claude Code를 사용하여 파이프라인을 구현한 뒤, 실제 transcript를 파이프라인에 투입하고 DB 추출 결과를 직접 확인하는 방식으로 품질을 검증했다. 단순히 코드를 생성하는 데 그치지 않고, 추출 결과를 transcript와 수작업으로 대조하여 문제를 발견하고 프롬프트를 반복 개선했다.

Gemini API 무료 티어 제약(모델당 일 20회)으로 인해 쿼터가 소진될 때마다 다른 모델로 교체해야 했고, 이 과정에서 의도치 않게 **모델 간 추출 품질 비교 실험**이 이루어졌다.

|---|---|---|---|---|---|---|

### 5.3 정성 평가

**컨텍스트 전달 반영 능력:**
- gemini-3.1-flash-lite: 이전 청크 액션 목록을 잘 반영해 중복 제거 효과 높음
- gemini-3.1-flash-lite-preview: 동일 프롬프트에서 컨텍스트를 덜 엄격하게 반영, 중복 재발

**한국어 회의 맥락 파악:**
- 모든 모델에서 날짜 추론(6/6 100%) 정확
- "제가 챙길게요" → 담당자 자동 매핑 : 모든 모델 성공
- "목요일 오후 그대로 가는 거니까" → 재확인 확정 표현: 모델마다 반영 여부 상이

**한계점:**
1. **모델 비일관성**: 동일 프롬프트에서 모델이 바뀌면 추출 건수 및 중복 여부가 달라짐
2. **Precision-Recall 트레이드오프**: 엄격한 추출 기준 → Precision↑ Recall↓, 느슨한 기준 → 반대
3. **청크 경계 문제**: 컨텍스트 전달로 완화되나 모델 능력에 의존하므로 완전 해소 불가
4. **단일 샘플 한계**: 회의 1건으로 설계된 confidence 룰과 추출 기준이 실제 다양한 회의에서도 동일하게 동작할지 미검증

### 5.4 도입 후 4주 운영·검증 계획

#### KPI 목표

| 지표 | 목표값 | 측정 방법 |
|---|---|---|
| Precision | ≥ 85% | 추출 건수 중 실제 액션아이템 비율 (주간 샘플링) |
| Recall | ≥ 75% | 실제 액션아이템 중 추출된 비율 (주간 샘플링) |
| 담당자 명시율 | ≥ 70% | assignee != null 비율 |
| 마감일 명시율 | ≥ 60% | deadline != null 비율 |
| 처리 시간 | ≤ 3분 | 회의 종료 → Slack 알림 수신까지 |

#### 주차별 계획

**1주차 — 데이터 수집 & 기준선 측정**
- 실제 회의 5건 파이프라인 적용
- 담당자가 직접 ground truth 레이블링
- 초기 Precision/Recall 측정 → 프롬프트 개선 방향 도출

**2주차 — 프롬프트 최적화**
- 오추출/누락 패턴 분석 후 추출 기준 정교화
- 모델 고정 (유료 플랜 전환으로 쿼터 제약 해소)
- confidence 룰 가중치 데이터 기반 조정

**3주차 — 대시보드·알림 검증**
- 실제 사용자(팀원) 피드백 수집
- Slack 알림 후 담당자 확인율·완료율 추적 시작
- confidence < 0.5 항목 검수 부담 측정

**4주차 — 안정화 & KPI 확정**
- 4주간 누적 데이터로 최종 KPI 측정
- confidence 임계값·추출 기준 최종 확정
- 100명 배포 전 edge case 목록 정리

#### 모니터링 항목 (운영 중 지속)

- confidence 분포 변화 (주간): 저신뢰 비율 증가 시 프롬프트 점검
- 모델 쿼터 현황: 소진 전 사전 알림 설정
- 중복 의심 항목 수 추이: 대시보드 ⚠️ 건수 주간 집계
- 액션아이템 완료율: `status = done` 전환 비율 추적
